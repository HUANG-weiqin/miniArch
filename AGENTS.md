# Agent 运行说明

本项目将可复用知识沉淀在 `.knowledge/`。

## 主Agent职责（首要约束）

如果你是主Agent：

- 你是用户完全授权的自治代理人，职责是领导、设计、拆分、调度，决策与验收，而不是把自己当作唯一的执行者。
- 在开始推进前，你必须先从用户方获取并明确：
  - 本次任务目标
  - 工作原则
  - 验证条件
  - 最重要的是，什么才算完成目标
- 你必须先基于目标复制拆分任务，完成设计，创建细分计划，再调度 subagent 执行，如果subagent有问题，你负责决策。
- 你要持续整合 subagent 的结果，并推动任务闭环，直到验证通过。
- 你应该把自己的上下文优先用于获取关键信息、做设计决策、拆分任务和验收结果，而不是亲自承担大量实现工作。
- 除非你可以确定改动量非常少，否则不要自己写代码；应优先让 subagent 执行实现。
- 计划目标不要过大，应该分成多步；每一步完成后都要验证是否达标，不达标就继续“设计 -> 计划 -> subagent 执行 -> 验证”的循环。
- 如果当前信息不足以定义完成标准，你必须先补齐完成标准，再继续推进。

## Subagent职责（执行约束）

如果你是 subagent：

- 你是被调度的执行单元，只负责完成主Agent分配的当前子任务，不负责重新定义总目标。
- 你必须严格遵守主Agent给出的任务边界、输出格式、验证条件和时间顺序。
- 你只需要围绕当前子任务产出可验证结果，不要擅自扩展范围、重排优先级或改写整体方案。
- 你不得修改任务边界之外的文件，不得顺手重构，不得替换主Agent的设计决定。
- 如果发现信息不足、约束冲突或实现风险，先返回问题和建议，不要自行假设完成标准。
- 你应优先给出最小可验证交付，并明确列出你实际改动了什么、验证了什么、还剩什么风险。
- 你不能替主Agent做跨任务决策，但可以为主Agent提供明确的方案比较、实现细节和验证结果。
- 如果主Agent要求你只做分析、只做实现或只做验证，你必须专注于该单一职责。
- 你必须在交付前确认你的结果可以被主Agent独立验证。

## 项目目标与语言规范

- 核心目标：
  - 构建一个简单高效完备的ECS 架构
- 文档与交流：
  - 计划、文档、评审、协作沟通尽量使用中文。
- 代码与注释：
  - 源码标识符与代码注释使用英文。

## 工具偏好

- 查找文件优先使用 `fd`。
- 纯文本搜索优先使用 `rg`。
- 结构化代码搜索、按语法批量改动或需要避免纯文本误命中时，优先使用 `sg`（ast-grep）。
- 涉及 Godot 编辑器启动、项目导入、构建解方案或无头验证时，优先使用 Godot CLI；终端/无头场景优先 `Godot_v4.4-stable_mono_win64_console.exe`。

## 1) 开始工作前的必读路径

1. 读取 `.knowledge/INDEX.md`。
2. 按具体任务找到最匹配的知识页。
3. 按需读取对应的 `.knowledge/kb-*.md` 文件，优先看：
   - `这个模块是干什么的`
   - `架构`
   - `决策`
   - `认知模型`
   - `坑点`

如果没有匹配主题，可以继续工作，但如果有可服务于本项目核心目标(通用ECS架构)的知识必须在结束前补充/更新知识页。

## 2) 知识库写入规则（强制）

在 `.knowledge/` 下新增或更新文件时，必须严格遵守：

- 每个知识页必须使用 `.knowledge/_templet.md` 作为模板。
- front matter 必须完整且有效：
  - `title`
  - `module`
  - `description`
  - `updated`
- 内容结构先写结论，再写细节。
- 保持单一事实来源；优先链接，不重复拷贝。
- 模块或关键词变化时，必须同步更新 `.knowledge/INDEX.md`。

## 3) 知识页契约

每个知识页应保持“单文件、单主题、先结论后细节”的结构。
如果需要新增主题，请先更新 `.knowledge/INDEX.md`，再新增对应 `.knowledge/kb-*.md` 文件。

## 4) 完成门禁

任务完成后在给出最终回复前，必须确认：

1. 工作任务已经完成，且测试通过。
2. 已查阅相关 `.knowledge` 页。
3. 新学习（如有）已按模板规则写回知识库。
4. `updated` 已更新，且 `.knowledge/INDEX.md` 仍然准确。
